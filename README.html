OllamaLens

A local, offline AI assistant and benchmarking platform for Small Language Models. Zero cloud. Zero API cost. Runs entirely on your own GPU.

No Cloud Required FastAPI + Python GTX 1660 Ti Optimised Streaming SSE Pydantic v2 SQLite Persistence

What is OllamaLens?

OllamaLens is a full-stack application that lets you interact with, benchmark, and compare local Large Language Models — all running on your own machine using Ollama.

💬
Chat Interface
Stream responses from Llama 3, Mistral, or Phi-3 in real time. Tokens appear as they're generated, just like ChatGPT.
📊
Benchmark Runner
Measure Time To First Token (TTFT) and Tokens Per Second (TPS) across N runs. Stored in SQLite for trend analysis.
⚔️
Model Comparison
Send the same prompt to all three models and see side-by-side performance metrics and response quality.
🔒
Structured JSON
Force the LLM to output schema-conformant JSON. Auto-retries with corrective prompts if validation fails.
📋
Technical Report
Auto-generated comparison report from all benchmark data. Exportable as JSON or Markdown.
🔌
Fully Offline
After pulling models once, the entire stack runs without internet. Your data never leaves your machine.

Hardware Constraints & Design Decisions

Hardware: Intel i5 9th Gen · 32 GB RAM · NVIDIA GTX 1660 Ti (6 GB VRAM)

Every technical decision traces back to the VRAM ceiling. Here's why:

VRAM is the binding constraint

Decision → q4_K_M Quantization A 7B model in FP16 precision requires ~14 GB VRAM — impossible on a 6 GB card. q4_K_M (4-bit K-quant medium) reduces this to ~4 GB with only ~2% quality degradation. The "K" in K_M means it uses a non-uniform quantization scheme that preserves the accuracy of "important" weights better than simple Q4.
PrecisionVRAM (7B model)QualityFits 1660 Ti?
FP16~14 GB100%❌ No
Q8~8 GB~99%❌ No
Q5_K_M~5.5 GB~97%⚠️ Barely
Q4_K_M ✓~4.1 GB~96%✅ Yes (headroom)
Q3~3.3 GB~91%✅ Yes (poor quality)

RAM (32 GB) — a strategic advantage

All three model files (~12 GB total) fit in RAM simultaneously. When you switch models, Ollama transfers the new model from RAM → VRAM in 3–8 seconds rather than reading from disk (which would take 30–60 seconds). The 32 GB RAM makes cold-start benchmarks more consistent.

Only one model in VRAM at a time

The /compare/run endpoint sends the same prompt to multiple models, but Ollama serialises GPU access automatically. The benchmark results reflect real-world single-user throughput.

Tech Stack — Every Tool Explained

Ollama

What: Local LLM runtime. Manages model downloads, CUDA inference, and exposes a REST API at localhost:11434.

Why not use llama.cpp directly? Ollama wraps llama.cpp and adds: automatic GPU detection, model version management, a stable REST API, and built-in JSON mode. We get identical performance with a maintained interface instead of managing C++ binaries and CUDA flags manually.

FastAPI

What: Modern Python async web framework. Auto-generates OpenAPI docs at /docs.

Why not Flask? Flask is synchronous. Each request occupies a thread. For SSE token streaming (one socket per active chat), Flask would need one thread per user. FastAPI is ASGI-based — one process can handle thousands of concurrent SSE streams on the event loop. No threading complexity, no race conditions.
Why not Django REST Framework? Django adds significant setup overhead (ORM migrations, settings files, app registry). For a focused API with ~8 endpoints, FastAPI delivers the same reliability in a fraction of the code. The built-in Pydantic integration means zero serialiser boilerplate.

Pydantic v2

What: Data validation library using Python type annotations. Every API request and response is a Pydantic model.

Why validate at the boundary? LLMs are probabilistic — they sometimes omit fields, change types, or return malformed data. Without Pydantic, a missing field causes an AttributeError deep in business logic with a cryptic stack trace. With Pydantic, invalid data is caught at the HTTP boundary and returns a structured 422 response with field-level error messages. Business logic never sees invalid data. 
# Example: if you POST {"model": "gpt-4", "prompt": "  "}, FastAPI returns:
{
  "detail": [
    {"loc": ["body", "model"], "msg": "value is not a valid enum member"},
    {"loc": ["body", "prompt"], "msg": "prompt must not be blank or whitespace only"}
  ]
}

httpx

What: Async HTTP client used to communicate with Ollama's REST API.

Why not `requests`? `requests` is synchronous. Calling it inside an async FastAPI route would block the entire event loop — no other requests could be handled while waiting for Ollama. httpx has an identical API but supports async/await, streaming response bodies, and HTTP/2 out of the box. 
# Separate timeouts for different failure modes:
OLLAMA_TIMEOUT = httpx.Timeout(
    connect=5.0,   # localhost connection: should be instant
    read=120.0,    # cold model + 500 tokens can take 30+ seconds
    write=10.0,
    pool=5.0,
)

Tenacity

What: Retry library with a decorator API. Handles exponential back-off, conditional retry, and logging.

Why not write retry loops manually? Manual retry adds ~30 lines of fragile, untested code. Tenacity encodes battle-tested retry behaviour in 4 decorator arguments. It also logs retry attempts automatically, which is invaluable for debugging. 
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=8),  # 2s → 4s → 8s
    retry=retry_if_exception_type((httpx.ReadTimeout, httpx.ConnectError)),
)
async def generate(self, model, prompt):
    ...

SQLModel + aiosqlite

What: SQLModel = Pydantic + SQLAlchemy ORM in a single class definition. aiosqlite = async SQLite driver.

Why SQLite instead of PostgreSQL? PostgreSQL requires a running server process, Docker, or a cloud subscription. SQLite is a single file — perfect for "zero infrastructure cost". For a local tool with fewer than 10,000 rows, SQLite is faster and simpler with no operational overhead.
Why SQLModel over raw SQLAlchemy? SQLModel unifies the Pydantic schema and the database table in one class. No risk of the API response schema drifting out of sync with the database columns. One source of truth. 
class BenchmarkRun(SQLModel, table=True):
    # This single class is BOTH the SQLite table AND the Pydantic response model
    id: Optional[int] = SQLField(default=None, primary_key=True)
    model: str
    tokens_per_second: float
    time_to_first_token_ms: float
    ...

jsonschema

What: JSON Schema validator (RFC draft-07).

Why not Pydantic for runtime JSON validation? Pydantic models require a static class definition at import time. For /chat/structured, the user supplies an arbitrary schema at runtime as a dict — we don't know the shape ahead of time. jsonschema.validate(instance, schema) works with any dict-format schema at runtime.

Async I/O with FastAPI

FastAPI uses Python's asyncio event loop. Every route function is async def:

# NON-blocking: while waiting for Ollama, the event loop
# handles other requests. No threads needed.
async def generate(request: ChatRequest):
    response = await client.generate(...)  # yields to event loop
    return response

This is the same model used by Node.js — one process, many concurrent I/O operations. On a machine where the GPU is the bottleneck (not the CPU), async is the correct choice because all the waiting is I/O waiting, not CPU waiting.

SSE Streaming

Server-Sent Events is a one-way channel from server to browser. It's how tokens appear in real time.

Why SSE and not WebSockets?

FactorSSEWebSockets
DirectionServer → Client onlyBidirectional
ProtocolPlain HTTPUpgrade handshake required
Auto-reconnect✅ Built-in❌ Manual
Browser supportEventSource API (native)WebSocket API (native)
ComplexityLowHigher
Good for token streaming?✅ Yes — one-wayOverkill

Wire Format

data: {"token": "Hello"}\n\n
data: {"token": " world"}\n\n
data: {"done": true, "eval_tokens": 42, "total_duration_ms": 1200}\n\n
data: [DONE]\n\n

Each \n\n flushes the chunk to the browser, triggering an onmessage event in the JavaScript EventSource API.

Pydantic v2 Validation

Every API boundary is guarded by a Pydantic model. Here's the full validation chain for a chat request:

HTTP Body arrives
→
ChatRequest.model_validate()
→
Enum check (model name)
→
Field validator (blank check)
→
Route handler runs

If any step fails, FastAPI returns HTTP 422 with structured error details before your code runs. This is the "fail fast" principle — detect errors as early as possible.

class ChatRequest(BaseModel):
    model: SupportedModel   # MUST be "llama3:8b...", "mistral:7b...", or "phi3:mini"
    prompt: str = Field(..., min_length=1, max_length=8000)

    @field_validator("prompt")
    @classmethod
    def prompt_not_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("prompt must not be blank or whitespace only")
        return v.strip()   # also strips whitespace as a side effect

Retry Logic with Tenacity

LLMs have two common transient failure modes on limited hardware:

  • ReadTimeout — model still loading into VRAM, takes longer than expected
  • ConnectError — Ollama daemon briefly unavailable

The retry decorator on OllamaClient.generate() handles both automatically:

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=8),
    retry=retry_if_exception_type((httpx.ReadTimeout, httpx.ConnectError)),
    before_sleep=before_sleep_log(logger, logging.WARNING),  # auto-logs
    reraise=True,  # if all attempts fail, re-raise the original exception
)
async def generate(self, model, prompt):
    ...

Why exponential back-off?

If the GPU is swapping memory, retrying immediately makes the problem worse. Exponential back-off gives the system time to recover:

2sAttempt 2 wait
4sAttempt 3 wait
raiseAfter 3 failures

Structured JSON Output Pipeline

The POST /chat/structured endpoint forces the LLM to return valid, schema-conformant JSON. Small quantised models occasionally emit malformed output — the pipeline handles this gracefully:

1. Append JSON instruction to system prompt
↓
2. Set format="json" in Ollama (constrains token sampling)
↓
3. Strip markdown fences from response (```json...```)
↓
4. json.loads() — parse to Python dict
↓
5. jsonschema.validate(dict, user_schema)
↓
6. If invalid: inject error into correction prompt, retry (max 3×)
↓
7. Return validated dict + attempt count

The attempts field in the response is a reliability signal — a model that consistently needs 3 attempts is less suitable for structured output tasks.

Benchmark Metrics Explained

TTFT — Time To First Token

Wall-clock milliseconds from when the HTTP request is sent to when the first non-empty token chunk arrives. This is the perceived "thinking time" — what the user experiences before seeing any output.

Measured with time.perf_counter() (sub-millisecond resolution, monotonic clock).

Why wall-clock, not Ollama's built-in timing? Ollama's eval_duration measures GPU compute time only — it excludes HTTP request serialisation, CUDA kernel warm-up, and token decode overhead. Wall-clock TTFT is what the user actually experiences, so it's the honest metric for a user-facing tool.

TPS — Tokens Per Second

eval_tokens / (total_wall_seconds). Higher is better. On a GTX 1660 Ti with q4_K_M models, expect 18–50 TPS depending on model size.

Why N runs instead of 1?

Run 1Cold (GPU cache miss)
Run 2Warm (typical)
Run 3Thermal throttle?

The median of 3 runs filters the cold-start outlier without requiring many runs. Median is preferred over mean because it's not affected by single extreme values.

How to Run

1. Install Ollama and Pull Models

# Install Ollama (Linux)
curl -fsSL https://ollama.com/install.sh | sh

# Pull models (each is 4+ GB — do one at a time)
ollama pull llama3:8b-instruct-q4_K_M
ollama pull mistral:7b-instruct-q4_K_M
ollama pull phi3:mini

# Start the Ollama daemon
ollama serve

2. Start the Backend

cd ollamalens/backend

# Create isolated Python environment
python -m venv .venv
source .venv/bin/activate        # Linux/Mac
# .venv\Scripts\activate         # Windows

pip install -r requirements.txt

# Copy environment config
cp ../.env.example .env

# Start the API server
uvicorn backend.main:app --reload --port 8000

3. Test the API

# Health check
curl http://localhost:8000/health

# Non-streaming chat
curl -X POST http://localhost:8000/chat/generate \
  -H "Content-Type: application/json" \
  -d '{"model":"phi3:mini","prompt":"What is recursion?","stream":false}'

# Run benchmark (3 iterations)
curl -X POST http://localhost:8000/benchmark/run \
  -H "Content-Type: application/json" \
  -d '{"model":"phi3:mini","runs":3}'

# Open interactive API docs
open http://localhost:8000/docs

Common Interview Questions

What is Pydantic and why did you use it?
Pydantic is a data validation library. I use it to validate every API request and response at the HTTP boundary, so invalid data never reaches the business logic. It turns runtime AttributeErrors into clear HTTP 422 responses with field-level error messages. In this project, it also serves double duty as the SQLite schema definition through SQLModel.
What is SSE? Why not WebSockets?
Server-Sent Events is a one-way HTTP channel from server to browser. Token streaming is strictly one-directional, so SSE is simpler and more appropriate than WebSockets. SSE works over plain HTTP with no protocol upgrade, browsers auto-reconnect on drops, and no extra library is needed on either end.
What is TTFT and why does it matter?
Time To First Token — how long until the user sees the first word of the response. Even if overall generation is slow, a low TTFT makes the interface feel responsive. It's a key UX metric for LLM products — OpenAI's API dashboard reports it for exactly this reason.
Why SQLite instead of PostgreSQL?
Zero infrastructure cost and zero setup. For a local tool with hundreds of rows, SQLite is faster to set up and operationally simpler. If this needed to scale to multiple users or concurrent writes, swapping to PostgreSQL is a one-line connection string change — SQLAlchemy abstracts the difference.
What is q4_K_M quantization?
Quantization reduces model weight precision from 16-bit floats to 4-bit integers, cutting VRAM usage by 4×. The K-quant variant uses non-uniform quantization that preserves the accuracy of "important" weights better than simple Q4. On a 6 GB GPU, it's the difference between running a 7B model at all or not — q4_K_M is the established sweet spot between speed, quality, and memory.
Why did you choose this set of models to compare?
Each model represents a different point in the speed/quality trade-off space: Llama 3 8B is Meta's strongest small model for instruction following; Mistral 7B is the open-source performance baseline many benchmarks reference; Phi-3 Mini is Microsoft's efficiency-focused model that punches above its weight for its size. Together they cover the realistic decision space for a developer choosing a local model.
How would you scale this to production?
Several changes: swap SQLite for PostgreSQL (connection string only), add authentication (FastAPI supports OAuth2 natively), containerise with Docker for reproducible deployment, add a Redis queue for benchmark jobs so long-running tasks don't block HTTP, and replace the Ollama process with a vLLM or TGI server for higher throughput. The FastAPI + Pydantic + async foundation handles all of these without architectural changes.
